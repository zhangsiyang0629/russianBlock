# 俄罗斯方块微信小游戏 - 事件系统需求规格

## 概述
本需求描述游戏中随机事件系统的设计架构、配置方式与触发逻辑。事件系统需具备良好的可扩展性，新增事件种类时无需修改核心调度代码。事件触发基于“砖块落地数量”而非时间，触发时机和事件本身均由配置驱动。

---

## 一、事件系统架构

### 核心设计原则
- **事件与关卡解耦**：每个关卡支持哪些事件、触发条件、触发概率均由配置决定。
- **易于扩展**：新增事件只需注册事件类型并实现对应逻辑，事件调度器无需修改。
- **事件种类标识**：每种事件使用唯一的 `int` 类型 ID 进行标识。

### 事件注册机制
- 维护一个全局事件注册表（如 `EventRegistry`），结构为 `Map<int, EventHandler>`。
- 新事件通过调用注册函数（如 `registerEvent(eventId, handler)`）加入注册表。
- 事件调度器根据配置中的事件 ID 列表，从注册表中获取对应处理逻辑并执行。

## 二、关卡事件配置

### 配置数据结构
每个关卡的事件配置包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `eventIds` | `int[]` | 该关卡支持触发的事件 ID 列表。为空列表 `[]` 表示无事件。特殊值 `[-1]` 表示所有已注册事件。 |
| `triggerBlocks` | `int[]` | 触发间隔配置列表，单位为“落地砖块数量”。例如 `[5, 8, 10]`。 |
| `triggerProbability` | `int` | 事件触发概率，取值范围 `0 ~ 100`。`100` 表示满足条件时必定触发，`0` 表示永不触发。 |

### 配置规则详解

#### 1. 事件列表 (`eventIds`)
- **空列表 `[]`**：该关卡不触发任何事件。
- **`[-1]`**：该关卡可触发所有已注册事件。
- **未配置**（即整个配置对象缺失或 `eventIds` 字段不存在）：视为 **全事件**，等效于 `[-1]`。
- **其他列表**：如 `[1, 3, 5]`，表示该关卡仅可触发 ID 为 1、3、5 的事件。

#### 2. 触发间隔列表 (`triggerBlocks`)
- 列表包含至少 1 个正整数，表示“每落地多少个砖块后尝试触发一次事件”。
- **运行机制**：
  1. 初始时，从 `triggerBlocks` 列表中**随机选取**一个值，作为当前触发间隔。
  2. 当累计落地砖块数达到该间隔时，尝试触发事件。
  3. 尝试完成后（无论是否成功触发），**再次从列表中随机选取**下一个触发间隔。
  4. 重复以上步骤，直至关卡结束。

> 示例：配置 `[5, 8, 10]`，初始随机到 `5` → 落地 5 个砖块后尝试触发 → 再次随机到 `10` → 落地 10 个砖块后尝试触发 → 再次随机...

#### 3. 触发概率 (`triggerProbability`)
- 当满足触发间隔条件时，以该概率决定是否真正触发事件。
- 随机判定：生成 `0 ~ 99` 的随机数，若小于 `triggerProbability`，则触发；否则不触发。
- 无论触发与否，均消耗本次触发机会，进入下一轮间隔计数。

---

## 三、事件触发流程

### 完整时序
1. 砖块落定，累计落地砖块计数 +1。
2. 判断累计数是否达到当前触发间隔。
   - 若未达到，无操作。
   - 若达到，执行步骤 3。
3. 从 `eventIds` 中随机选取一个事件 ID（若为 `[-1]` 或未配置，则从所有已注册事件中随机选取）。
4. 生成随机数，判断是否通过概率检查。
   - 若未通过，无事件触发，重置计数，重新选取间隔。
   - 若通过，执行步骤 5。
5. 调用对应事件 ID 的处理函数，执行事件逻辑。
6. 重置累计落地砖块计数为 0。
7. 从 `triggerBlocks` 列表中重新随机选取下一次触发间隔。
