# 俄罗斯方块微信小游戏 - 排行榜功能需求规格

## 概述
本需求描述游戏内排行榜功能的交互流程、界面设计与数据存储方案。核心原则：**排行榜入口与隐私授权流程完全解耦，无论用户是否同意隐私政策或头像授权，均可正常查看排行榜内容**。排行榜数据采用分页加载，每次仅获取10条，支持用户下滑动态加载更多。

---

## 排行榜类型

游戏提供两个独立的排行榜：

| 排行榜名称 | 排序字段 | 排序规则 | 展示上限 |
|------------|----------|----------|----------|
| **最高分数排行榜** | `highScore` | 降序（分数越高排名越靠前） | 前 **3000** 名 |
| **最高关卡排行榜** | `level` | 降序（关卡越高排名越靠前） | 前 **3000** 名 |

> 若分数/关卡相同，则按 `updateTime` 升序排列（较早达成的玩家排名靠前）。

## UI 界面设计

### 主界面入口按钮
- 游戏封面底部已经有绘制了scores入口按钮，只是点击没有反应，现在要用起来，点击后显示最高分数排行榜。
- 游戏封面底部新增1个入口按钮：
  - **Levels 按钮**：展示文案「**关卡榜**」或「**Levels**」，点击后显示最高关卡排行榜。
- 按钮样式需与现有 UI 风格一致，具体视觉参考 `ui/rank.html` 设计稿。

### 排行榜面板
- 点击入口按钮并完成必要的授权流程后，弹出半屏排行榜面板（或独立场景，根据设计稿决定）。
- 面板顶部包含两个标签页（或切换按钮）：
  - **分数榜** 标签
  - **关卡榜** 标签
- 榜单列表展示内容：
  - **排名**（1 ~ 3000）
  - **玩家头像**（若未授权则显示默认头像）
  - **玩家昵称**（若未授权则显示默认游客昵称）
  - **分数 / 关卡数**
- 列表支持**下滑滚动**，当滚动到底部时自动加载下一页数据（每次10条）。

## 交互流程（隐私授权与排行榜解耦）

### 核心规则
- 用户**点击排行榜入口按钮**时，首先检查授权状态：
  1. **检查隐私授权**：调用 `wx.getPrivacySetting` 判断用户是否已同意隐私政策。
  2. **检查头像授权**：若隐私已同意，再判断是否已获取用户真实头像/昵称（通过查询云数据库或本地标记）。
- **授权流程优先于排行榜打开**：
  - 若需要隐私授权，则触发隐私授权弹窗，等待用户操作（同意或拒绝）。
  - 若隐私已同意但未授权头像，则显示头像授权按钮（不阻塞流程，但排行榜打开前可引导用户授权）。
  - **所有授权交互完成后（无论用户同意还是拒绝），再打开排行榜面板**。
- **关键交互承诺**：
  - 用户点击排行榜入口后，**仅需一次点击**即可最终看到排行榜内容（授权流程自动衔接，无需用户二次点击入口）。
  - 用户拒绝隐私授权或拒绝头像授权，排行榜依然正常打开（使用默认游客信息）。

### 流程时序（以首次点击「分数榜」为例）

1. 用户点击 **「Scores」** 按钮。
2. **第一步：检查隐私授权状态**
   - 调用 `wx.getPrivacySetting` 获取当前隐私同意状态。
   - 若 `needAuthorization` 为 `true`，则：
     - 调用 `wx.requirePrivacyAuthorize` 触发隐私授权流程。
     - 弹出隐私授权弹窗，等待用户点击「同意」或「不同意」。
     - 无论用户选择什么，记录操作结果，继续后续流程。
3. **第二步：检查头像授权状态**（仅当隐私已同意或用户刚同意时）
   - 查询本地缓存或云数据库，判断当前用户是否已有真实头像/昵称（非默认游客信息）。
   - 若**无**真实信息，则：
     - 创建 `wx.createUserInfoButton` 按钮，引导用户授权获取微信头像昵称。
     - 等待用户操作（点击授权、取消或忽略）。
     - **注意**：此步骤不阻塞排行榜打开，用户可点击按钮授权，也可直接忽略。若用户忽略，仍可继续。
4. **第三步：打开排行榜面板**
   - 以上检查与交互完成后，立即打开排行榜面板（半屏或独立场景）。
   - 面板显示加载中状态，同时调用云函数拉取第一页数据（前10名）。
5. **第四步：渲染排行榜**
   - 数据返回后，渲染列表。
   - 用户可下滑加载更多（每次10条），直至加载完3000条或全部数据。

### 用户拒绝后的再次触发
- 若用户曾经拒绝隐私授权，后续再次点击排行榜入口时，仍会重新触发隐私授权弹窗（微信平台控制，开发者无需特殊处理）。
- 每次点击入口都会遵循上述时序，确保授权状态最新。

## 分页加载规格

### 请求参数
每次向云函数请求排行榜数据时，需传递：
- `type`：`'score'` 或 `'level'`，表示榜单类型。
- `skip`：跳过的记录数，用于分页（首次为 `0`，后续为当前已加载数量）。
- `limit`：固定为 `10`。

### 响应数据
云函数返回：
- `data`：包含 `nickname`、`avatarUrl`、`highScore` / `level` 等字段的数组。
- `hasMore`：布尔值，表示是否还有更多数据（`skip + limit < 3000` 且未到数据尾端）。

### 客户端处理
- 首次加载 `skip = 0`，清空现有列表。
- 下滑到底部时，若 `hasMore` 为 `true`，则用当前已加载数量作为 `skip` 请求下一页，并将新数据追加到列表末尾。
- 若已加载数量达到 3000 或 `hasMore` 为 `false`，则停止加载，可展示「已加载全部榜单」提示。

## 数据表设计

### 说明
由于排行榜需要高效的排序查询，且两个榜单排序字段不同，**建议建立独立的排行榜数据表**，与基础玩家数据表 `player_data` 分离。两个排行榜可共用同一张表，通过索引优化查询。

### 表结构：`rankings`

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `_openid` | String | 是 | 用户唯一标识，关联玩家数据。 |
| `nickname` | String | 是 | 玩家昵称（游客或授权后昵称）。 |
| `avatarUrl` | String | 是 | 玩家头像URL（默认或授权后头像）。 |
| `highScore` | Number | 是 | 历史最高分，用于分数榜排序。 |
| `level` | Number | 是 | 最高关卡数，用于关卡榜排序。 |
| `updateTime` | Date | 是 | 记录更新时间，用于同分/同关卡时次级排序。 |

### 数据同步策略
- 每当玩家完成一局游戏，或通关新关卡时，通过云函数更新 `player_data` 表，并**同步更新** `rankings` 表中该玩家的 `highScore` / `level` 及 `updateTime`。
- 若玩家未授权头像/昵称，使用默认游客信息写入。

### 索引设计（至关重要）

为支持高效分页排序查询，需在云开发控制台为 `rankings` 集合创建以下索引：

| 索引名称 | 字段 | 排序方向 | 用途 |
|----------|------|----------|------|
| `idx_score_time` | `highScore` 降序，`updateTime` 升序 | 复合索引 | 最高分数排行榜查询 |
| `idx_level_time` | `level` 降序，`updateTime` 升序 | 复合索引 | 最高关卡排行榜查询 |
