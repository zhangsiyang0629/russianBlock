# 俄罗斯方块微信小游戏 - 隐私授权交互需求规格

## 概述
本需求描述游戏中隐私授权与用户信息获取的交互流程，确保合规且用户体验流畅。核心原则：**游戏核心玩法（点击Play开始游戏）与隐私授权、头像授权完全解耦，用户无需授权即可正常游戏。**

---

## UI 元素说明

| 元素 | 位置 | 初始状态 | 动态行为说明 |
|------|------|----------|--------------|
| 隐私条款勾选框 | 主界面/设置界面 | 默认未勾选 | 用户同意隐私条款后，**该控件从界面永久移除**，不再显示。 |
| 隐私授权弹窗 | 浮层 | 隐藏 | 原生弹窗，展示隐私政策摘要，包含「同意」与「不同意」按钮。 |
| 头像授权按钮 | 主界面/排行榜入口附近 | 默认隐藏 | 由 `wx.createUserInfoButton` 创建的原生按钮。**仅当用户同意隐私条款后创建**。若用户未点击授权，下次进入游戏时仍会显示。 |
| Play 按钮 | 主界面中央 | 可见 | 点击后直接开始游戏，无任何前置授权检查。 |

## 交互流程

### 1. 隐私条款勾选框点击行为
- **触发条件**：用户点击「隐私条款」勾选框。
- **行为逻辑**：
  1. 立即弹出**隐私授权弹窗**（自定义样式）。
  2. 弹窗应清晰展示隐私政策核心内容，并提供「同意」与「不同意」按钮。
  3. **注意**：此时复选框**不立即改变勾选状态**，待用户操作弹窗后再处理。

### 2. 隐私授权弹窗用户操作

#### 情况 A：用户点击「同意」
- **后续行为**：
  1. 调用 `privacyResolve({ event: 'agree' })` 完成隐私授权流程。
  2. 关闭弹窗。
  3. **移除**「隐私条款勾选框」控件，该控件从界面上永久消失。
  4. **自动创建**「头像授权按钮」（调用 `wx.createUserInfoButton`），引导用户授权获取头像昵称。
  5. 记录用户已同意隐私的状态（本地存储或云端标记），确保下次启动不再显示隐私相关控件。

#### 情况 B：用户点击「不同意」
- **后续行为**：
  1. 调用 `privacyResolve({ event: 'disagree' })` 告知平台用户拒绝。
  2. 关闭弹窗。
  3. **保持**「隐私条款」勾选框为**未勾选**状态，控件保留。
  4. 不创建头像授权按钮，游戏仍以游客身份运行。

### 3. 头像授权按钮交互
- **按钮创建时机**：仅当用户完成隐私授权同意后自动创建。
- **按钮显示规则**：
  - 首次同意隐私后立即创建。
  - 若用户未点击授权直接开始游戏，**按钮保留在界面上**，不会自动消失。
  - 用户关闭游戏后再次进入时，若仍未授权过头像，**头像授权按钮应再次显示**。
  - 一旦用户成功授权并获取到头像昵称，按钮永久销毁，不再显示。
- **点击按钮后**：调用微信原生授权，获取用户头像、昵称的临时 URL。
- **成功获取后**：
  - 将临时头像上传至云存储，获取永久 `fileID`。
  - 通过云函数将用户信息（`nickname`、`avatarUrl`）更新到云数据库 `player_data` 集合中（基于 `_openid`）。
  - 销毁该按钮（不再显示）。
- **若用户拒绝或失败**：按钮保留，用户可再次点击；不影响游戏进行。

### 4. Play 按钮行为
- **触发条件**：用户点击「Play」按钮。
- **行为逻辑**：
  - **直接进入游戏主玩法场景**，无任何授权检查或前置条件。
  - 游戏过程中以游客身份运行（若未授权头像）或已授权身份运行。
  - 体力扣除、分数记录等逻辑正常执行，与授权状态无关。

## 状态持久化说明

| 状态项 | 存储位置 | 用途 |
|--------|----------|------|
| 隐私条款是否已同意 | 本地存储 + 云端 `player_data`（建议） | 判断是否移除隐私勾选框；下次启动时跳过隐私授权流程。 |
| 头像是否已授权 | 云端 `player_data` 中是否存在该用户的 `nickname` / `avatarUrl` 字段 | 判断是否显示头像授权按钮；若已存在则不再创建按钮。 |

> 建议：每次启动时通过云数据库查询用户信息是否存在，若不存在且隐私已同意，则显示头像授权按钮。

## 边界情况处理

| 场景 | 处理方式 |
|------|----------|
| 用户同意隐私，但从未点击头像授权按钮，关闭游戏后再次进入 | 隐私勾选框不再显示；**头像授权按钮重新显示**，引导用户授权。 |
| 用户同意隐私并授权头像后，在微信设置中取消授权 | 游戏下次启动时，云数据库中仍存有旧头像信息，但实际授权已失效。此时可选择性引导用户重新授权（非必须）。 |
| 用户拒绝隐私后，再次点击勾选框 | 重新弹出隐私授权弹窗，流程同上。 |
| 头像授权按钮已创建但用户未点击，直接点击Play开始游戏 | 允许；游戏正常运行，按钮保留在界面上。 |

## 技术实现要点

### 隐私授权监听与弹窗
- 在 `game.js` 中注册 `wx.onNeedPrivacyAuthorization` 监听。
- 弹窗触发时保存 `resolve` 函数，在用户点击同意/不同意时调用。

### 头像按钮创建
- 使用 `wx.createUserInfoButton` 创建原生按钮。
- 按钮样式需符合游戏 UI 风格，位置建议在排行榜入口或设置界面。
- 每次启动时检查云端是否已有用户信息，若无且隐私已同意，则创建按钮。

### 游客模式默认值
- 若用户未授权头像昵称，使用默认规则：
  - 昵称：`游客` + 毫秒时间戳 + 6位随机数
  - 头像：内置默认头像图片

## 界面示意
1. **首次进入**：主界面包含「Play」按钮、「隐私条款」勾选框（未勾选）。
2. **点击勾选框** → 弹出隐私授权弹窗。
3. **点击「同意」** → 勾选框消失，界面出现「获取头像昵称」按钮。
4. **用户可直接点击 Play 开始游戏**，头像按钮保留。
5. **下次启动**：若用户仍未授权头像，头像按钮再次出现；若已授权，则不再出现。
