# 俄罗斯方块微信小游戏 - 音乐播放需求规格

## 概述
本需求描述游戏中背景音乐的存储、获取与播放逻辑。音乐资源采用双源容灾策略，优先从微信云存储获取，失败时回退至腾讯云对象存储（COS）。播放逻辑支持动态扩展，无需因音乐数量变化而修改代码。

## 音乐加载策略

### 双源容灾优先级
1. **优先**：尝试从微信云存储加载音乐。
2. **回退**：若云存储加载失败（如网络异常、文件不存在），则尝试从腾讯云 COS 加载。
3. **兜底**：若 COS 也加载失败，且当前尝试播放的是 `music_1`，则**不播放任何音乐**（静音状态）。
4. **同名回退**：若加载的非 `music_1` 的音乐失败，则**不再尝试加载 `music_1` 作为替代**，直接静音处理。下一局开始时，再重新随机选择。

### 关键说明
- 加载失败仅影响当前局音乐播放，下一局游戏开始时会重新随机选择并尝试加载。
- 不设默认强制播放的兜底音乐，避免用户反复听到同一首而产生疲劳感。

## 动态音乐列表管理

### 核心原则
- **代码不写死音乐数量**，管理人员新增或删除音乐文件后，无需修改前端代码。

### 实现方案
采用**数字递增探测法**，在游戏启动时自动探测可用的音乐列表：

1. 从 `x = 1` 开始，依次尝试加载 `music_x.mp3`。
2. 若加载成功（HTTP 状态码 200），则将该音乐加入可用列表，并继续探测 `x + 1`。
3. 若连续多个序号（如连续 3 个）加载失败，则认为已到达列表末尾，停止探测。
4. 探测完成后，生成当前可用的音乐列表 `availableMusicList`。

### 探测时机
- 游戏**首次启动时**执行探测，结果缓存于内存中。
- 也可在用户进入设置页面时手动触发重新探测（可选）。

### 边界处理
- 若所有音乐均加载失败，`availableMusicList` 为空，则不播放任何背景音乐。
- 管理员需确保音乐文件命名连续（`music_1, music_2, ...`），中间不留空号。

## 播放规则

### 触发时机
- 用户点击「Play」开始游戏后，**进入关卡时**开始播放音乐。

### 选曲规则
1. 每一局游戏开始时，从 `availableMusicList` 中**随机选取一首**。
2. 选中的音乐**单曲循环播放**，直至本局游戏结束。
3. 游戏结束（死亡/通关结算）后，停止当前音乐。
4. 下一局重新开始时，再次随机选取一首。

### 暂停/恢复
- 游戏暂停（点击暂停按钮）时，音乐暂停。
- 游戏继续时，音乐从暂停位置恢复播放。

## 技术实现要点

### 播放器实例
- 使用 `wx.createInnerAudioContext()` 创建音频实例。
- 设置 `loop = true` 实现单曲循环。
- 设置 `obeyMuteSwitch = false`，确保在用户开启静音键时仍能播放（游戏场景适用）。
